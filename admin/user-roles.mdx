## Overview

Safaris Web uses a simple authentication-based permission system. The current implementation focuses on two main access levels: authenticated users and guests.

## Access Levels

### Guest Users (Unauthenticated)

Guest users have access to public-facing content:

**Allowed Actions:**
- View published tours (`/safaris`, `/tours/{category}`)
- Browse blog posts (`/blog`, `/blogs/show/{id}`)
- Watch videos (`/videos`)
- Submit enquiries (`/enquiries/create`, `/enquiries/store`)
- View static pages (about, contact, FAQs, etc.)
- Search for tours (`/search`)

**Restricted Actions:**
- Cannot access admin dashboard
- Cannot create, edit, or delete content
- Cannot manage bookings
- Cannot view user profiles

### Authenticated Users (Logged In)

Authenticated users must have a verified email to access most admin functionality.

**Allowed Actions:**
- Access admin dashboard (`/admin`, `/dashboard`)
- View their own profile (`/profile`)
- View all content in admin mode

**Required for Content Management:**

All create, update, delete operations require `['auth', 'verified']` middleware.

## User Model

The User model implements email verification:

```php
use Illuminate\Contracts\Auth\MustVerifyEmail;

class User extends Authenticatable implements MustVerifyEmail
{
    protected $fillable = [
        'name',
        'email',
        'password',
    ];

    protected $hidden = [
        'password',
        'remember_token',
    ];

    protected $casts = [
        'email_verified_at' => 'datetime',
    ];

    public function bookings()
    {
        return $this->hasMany('App\\Booking');
    }
}
```

### User Attributes

| Attribute | Type | Description |
|-----------|------|-------------|
| `id` | integer | Unique user identifier |
| `name` | string | User's full name |
| `email` | string | User's email address (unique) |
| `email_verified_at` | datetime | Timestamp when email was verified |
| `password` | string | Hashed password |
| `remember_token` | string | Token for "remember me" functionality |
| `created_at` | datetime | Account creation timestamp |
| `updated_at` | datetime | Last account update timestamp |

### User Relationships

**Bookings:** A user can have many bookings

```php
$user->bookings(); // Returns all bookings created by this user
```

## Middleware Protection

Safaris Web uses Laravel's built-in middleware for access control:

### Auth Middleware

Protects routes that require authentication:

```php
->middleware('auth')
```

**Applied to:**
- Admin dashboard (`/admin`, `/dashboard`)
- User profile (`/profile`)
- All content management routes

**Behavior:** Redirects unauthenticated users to login page

### Verified Middleware

Requires email verification for sensitive operations:

```php
->middleware(['auth', 'verified'])
```

**Applied to:**
- Creating tours, blogs, videos, categories
- Editing tours, blogs, videos, categories
- Deleting tours, blogs, videos, categories
- Publishing/unpublishing content
- Managing bookings
- Viewing bookings list

**Behavior:** Redirects unverified users to `/email/verify`

## Checking User Status

You can check user authentication and verification status in your application:

### Check if User is Authenticated

```php
use Illuminate\Support\Facades\Auth;

if (Auth::check()) {
    // User is logged in
    $user = Auth::user();
}
```

### Check if Email is Verified

```php
if (Auth::user()->hasVerifiedEmail()) {
    // User has verified their email
}
```

### Get Current User

```php
$user = Auth::user();
$userId = Auth::id();
$userName = Auth::user()->name;
$userEmail = Auth::user()->email;
```

## Implementing Role-Based Access Control (RBAC)

<Note>
  The current implementation does **not** include role-based access control. All verified users have the same permissions.
</Note>

To implement RBAC, consider adding:

<Steps>
  <Step title="Add Role Field to Users Table">
    Create a migration to add a `role` column:
    ```php
    Schema::table('users', function (Blueprint $table) {
        $table->string('role')->default('user');
    });
    ```
  </Step>
  
  <Step title="Create Middleware for Role Checking">
    Create a custom middleware to check user roles:
    ```php
    php artisan make:middleware CheckRole
    ```
  </Step>
  
  <Step title="Apply Role Middleware to Routes">
    Protect routes with role-based middleware:
    ```php
    Route::middleware(['auth', 'verified', 'role:admin'])
        ->group(function () {
            // Admin-only routes
        });
    ```
  </Step>
  
  <Step title="Update User Model">
    Add helper methods to check roles:
    ```php
    public function isAdmin() {
        return $this->role === 'admin';
    }
    ```
  </Step>
</Steps>

## Security Recommendations

<Warning>
  **Important Security Considerations:**
  
  1. User management routes (`/users/*`) are not protected by authentication middleware
  2. Enquiry routes are publicly accessible, which may be intentional but should be reviewed
  3. Consider implementing role-based access control for better permission management
  4. Add authorization policies to check if users can edit/delete specific resources
</Warning>
