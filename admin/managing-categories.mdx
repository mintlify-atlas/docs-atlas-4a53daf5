## Overview

The Category Management system allows you to organize tours into hierarchical categories and subcategories. Categories help users browse and filter tours by type, destination, or activity. All category operations are handled by the `CategoryController`.

**Controller:** `App\Http\Controllers\CategoryController`  
**Model:** `App\Models\Category`  
**Table:** `categories`  
**Route Prefix:** `/category`

<Note>
  All category management operations require `['auth', 'verified']` middleware - you must be logged in and have a verified email.
</Note>

## Category Routes

| Action | Route | Name | Method | Middleware |
|--------|-------|------|--------|------------|
| List all categories | `/categories` | `categories` | GET | None |
| Create category form | `/category/create` | `category.create` | GET | `['auth', 'verified']` |
| Store new category | `/category/new` | `category.store` | POST | `['auth', 'verified']` |
| Edit category form | `/category/edit/{id}` | `category.edit` | GET | `['auth', 'verified']` |
| Update category | `/category/update/{id}` | `category.update` | POST | `['auth', 'verified']` |
| Delete category | `/category/delete/{id}` | `category.delete` | DELETE | `['auth', 'verified']` |

## Category Model

The Category model uses a simple table structure:

```php
protected $table = "categories";
```

<Note>
  The model does not define fillable attributes, which means all fields are mass-assignable. This is a security concern.
</Note>

### Database Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | integer | Auto | Unique category identifier |
| `category_name` | string | Yes | Main category name |
| `sub_category` | string | No | First subcategory |
| `sub_category1` | string | No | Second subcategory |
| `sub_category2` | string | No | Third subcategory |
| `created_at` | datetime | Auto | Creation timestamp |
| `updated_at` | datetime | Auto | Last update timestamp |

**Category Structure:**
- **category_name** - Main category (e.g., "Wildlife Safari", "Beach Holiday")
- **sub_category** - Primary subcategory (e.g., "Big Five Safari")
- **sub_category1** - Secondary subcategory (e.g., "Luxury Safari")
- **sub_category2** - Tertiary subcategory (e.g., "Photography Safari")

public function index()
{

    return view('categories.index', compact('categories'))

### Store New Category

**Route:** `/category/new`  
**Method:** POST  
**Controller Method:** `store(Request $request)`  
*Source: CategoryController.php:37-53*

#### Validation Rules

```php
$request->validate([
    'category_name' => 'required'
]);
```

**Required Fields:**
- `category_name` - Main category name (required)

**Optional Fields:**
- `sub_category` - First subcategory
- `sub_category1` - Second subcategory
- `sub_category2` - Third subcategory

#### Storage Logic

```php
$category = new Category;

$category->category_name = $request->category_name;
$category->sub_category = $request->sub_category;
$category->sub_category1 = $request->sub_category1;
$category->sub_category2 = $request->sub_category2;

$category->save();
```
*Source: CategoryController.php:43-50*

<Note>
  Subcategories are stored even if empty/null. The database should allow NULL values for these fields.
</Note>

#### Success Response

After successful creation:

```php
return redirect()->route('categories')
    ->with('success', 'Category Saved!');
```

## Deleting a Category

**Route:** `/category/delete/{id}`  
**Method:** DELETE  
**Controller Method:** `destroy($id)`  
*Source: CategoryController.php:109-116*

```php
public function destroy($id)
{
    $category = Category::find($id);
    $category->delete();
    
    return redirect()->route('categories')
        ->with('success', 'Deleted Successfully');
}
```

<Warning>
  **Deletion is permanent!** There is no soft delete implemented. Deleting a category may affect associated tours.
</Warning>

<Steps>
  <Step title="Check for Associated Tours">
    Before deleting, verify no tours are using this category
  </Step>
  
  <Step title="Locate the Category">
    Find the category you want to delete in the categories list at `/categories`
  </Step>
  
  <Step title="Confirm Deletion">
    Click the delete button (should show confirmation dialog)
  </Step>
  
  <Step title="Category Removed">
    The category is permanently deleted from the database
  </Step>
</Steps>

<Note>
  Deleting a category does **not** delete associated tours. Tours will still reference the deleted category name, which may cause issues.
</Note>

## Best Practices

<AccordionGroup>
  <Accordion title="Naming Conventions" icon="font">
    - Use clear, descriptive names
    - Be consistent with capitalization (Title Case recommended)
    - Avoid abbreviations unless widely understood
    - Keep names concise (2-4 words)
    - Use singular or plural consistently
    
    **Good examples:**
    - Wildlife Safari
    - Beach Holiday
    - Cultural Experience
    - Mountain Trekking
    
    **Avoid:**
    - WL Safari (unclear abbreviation)
    - beach holidays and relaxation packages (too long)
    - WILDLIFE (all caps)
  </Accordion>
  
  <Accordion title="Hierarchy Planning" icon="sitemap">
    - Plan your category structure before creating
    - Keep hierarchy shallow (2-3 levels max)
    - Ensure subcategories logically belong to parent
    - Don't duplicate information across levels
    - Consider how users will navigate
    
    **Example structure:**
    ```
    Wildlife Safari (main)
      └─ Big Five Safari (sub_category)
      └─ Bird Watching (sub_category1)
      └─ Night Safari (sub_category2)
    ```
  </Accordion>
  
  <Accordion title="Category Management" icon="tasks">
    - Create all major categories before adding tours
    - Review and consolidate duplicate categories
    - Don't delete categories with associated tours
    - Update tour associations before deleting categories
    - Keep the category list manageable (10-20 main categories)
  </Accordion>
  
  <Accordion title="Subcategory Usage" icon="layer-group">
    - Use subcategories for meaningful distinctions
    - Don't create subcategories for minor variations
    - Consider if subcategory needs its own category
    - Leave subcategories empty if not needed
    - Use all three subcategory fields only when necessary
  </Accordion>
</AccordionGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Duplicate Categories" icon="clone">
    **Problem:** Multiple categories with similar names
    
    **Examples:**
    - "Wildlife Safari" and "Wildlife Safaris"
    - "Beach Holiday" and "Beach Holidays"
    
    **Solution:**
    1. Decide on a consistent naming convention
    2. Update all tours using the duplicate category
    3. Delete the duplicate category
    4. Standardize future entries
  </Accordion>
  
  <Accordion title="Cannot Delete Category" icon="lock">
    **Problem:** Tours still reference the category
    
    **Solution:**
    1. Find all tours using this category:
       ```php
       $tours = Tours::where('category', 'Old Category')->get();
       ```
    2. Update each tour to use a different category
    3. Then delete the unused category
    
    **Better approach:** Implement foreign key constraints with `onDelete('restrict')`
  </Accordion>
  
  <Accordion title="Tours Not Filtering Correctly" icon="filter">
    **Problem:** Tours don't appear when filtering by category
    
    **Possible causes:**
    - Category name doesn't match exactly (case-sensitive)
    - Tours use old category name
    - Tours have typos in category field
    
    **Solution:**
    - Ensure exact name match between categories and tours
    - Update tours to use correct category names
    - Consider using category IDs instead of names (requires refactoring)
  </Accordion>
  
  <Accordion title="Missing Categories in Tour Form" icon="exclamation-triangle">
    **Problem:** Category dropdown is empty when creating tours
    
    **Causes:**
    - No categories created yet
    - Categories query failing
    - View not rendering correctly
    
    **Solution:**
    - Create at least one category first
    - Check controller passes `$categories` to view
    - Verify view template renders dropdown correctly
  </Accordion>
</AccordionGroup>

## Security Considerations

<Warning>
  **Security Issues:**
  
  1. **No fillable array defined** - All fields are mass-assignable
  2. **No authorization checks** - Any verified user can manage categories
  3. **No validation on subcategories** - Can store arbitrary data
  4. **No foreign key constraints** - Tours can reference non-existent categories
</Warning>

### Recommended Improvements

<Steps>
  <Step title="Add Fillable Array">
    Define which fields can be mass-assigned:
    
    ```php
    // In Category model
    protected $fillable = [
        'category_name',
        'sub_category',
        'sub_category1',
        'sub_category2'
    ];
    ```
  </Step>
  
  <Step title="Implement Authorization">
    Check if user can manage categories:
    
    ```php
    // Create a policy
    php artisan make:policy CategoryPolicy
    
    // In controller
    $this->authorize('manage', Category::class);
    ```
  </Step>
  
  <Step title="Add Unique Validation">
    Prevent duplicate category names:
    
    ```php
    $request->validate([
        'category_name' => 'required|unique:categories,category_name',
    ]);
    ```
  </Step>
  
  <Step title="Use Foreign Keys in Tours">
    Reference category by ID instead of name:
    
    ```php
    // Migration
    $table->foreignId('category_id')
          ->constrained()
          ->onDelete('restrict');
    ```
  </Step>
</Steps>
