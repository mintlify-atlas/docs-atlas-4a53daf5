## Overview

The Bookings Management system allows you to view and manage customer tour bookings, confirm reservations, and track booking details. All booking management operations are handled by the `BookingController`.

**Controller:** `App\Http\Controllers\BookingController`  
**Model:** `App\Models\Booking`  
**Table:** `tour_bookings`  
**Route Prefix:** `/bookings`

<Note>
  All booking management operations require `['auth', 'verified']` middleware - you must be logged in and have a verified email.
</Note>

## Booking Routes

| Action | Route | Name | Method | Middleware |
|--------|-------|------|--------|------------|
| List all bookings | `/bookings` | `bookings` | GET | `['auth', 'verified']` |
| View booking details | `/bookings/show/{id}` | `bookings.show` | GET | `['auth', 'verified']` |
| Create booking form | `/bookings/create/{destination}` | `bookings.create` | GET | None |
| Store new booking | `/bookings/new` | `bookings.store` | POST | `['auth', 'verified']` |
| Edit booking form | `/bookings/edit/{id}` | `bookings.edit` | GET | `['auth', 'verified']` |
| Update booking | `/bookings/update/{id}` | `bookings.update` | POST | `['auth', 'verified']` |
| Delete booking | `/bookings/delete/{id}` | `bookings.delete` | DELETE | `['auth', 'verified']` |
| Confirm booking | `/bookings/publish/{id}` | `bookings.confirm` | DELETE | `['auth', 'verified']` |

<Warning>
  The confirm route uses DELETE method (should probably be PUT/PATCH). This may be a typo in the route definition.
</Warning>

## Viewing Bookings

### List All Bookings

**Route:** `/bookings`  
**Method:** GET  
**Controller Method:** `index()`  
*Source: BookingController.php:21-26*

```php
public function index()
{
    $bookings = Booking::latest()->paginate(10);
    return view('bookings.index', compact('bookings'))
```

**Route:** `/bookings/new`  
**Method:** POST  
**Controller Method:** `store(Request $request)`

```php
$request->validate([
```

**Required Fields:**

- Booking details: arrival_date, adults, kids, destination

- Security: g-recaptcha-response (reCAPTCHA)

#### Email Confirmation Logic

```php
if($request->email1 == $request->email2) {
    $email = $request->email1;
} else {

        ->with('failure', 'please provide matching emails');
}
```

**Email Validation:**
- User must enter email twice (email1 and email2)
- Both entries must match exactly
- If they don't match, booking is rejected

## Confirming Bookings

### Confirm/Unconfirm Booking

**Route:** `/bookings/publish/{id}`  
**Method:** DELETE (should be PUT/PATCH)  
**Controller Method:** `confirm($id)`  
*Source: BookingController.php:168-180*

```php
public function confirm($id)
{
    $booking = Booking::find($id);
    
    if ($booking->status == 0) {
        $booking->status = 1;  // Confirm
        $booking->update();
    } else {
        $booking->status = 0;  // Unconfirm
        $booking->update();
    }
    
    return redirect()->route('bookings')
        ->with('success', 'Booking deleted successfully!');
}
```

**Booking Status:**
- `status = 0` - Pending (awaiting confirmation)
- `status = 1` - Confirmed (booking approved)

<Warning>
  The success message says "Booking deleted successfully!" but the method actually toggles confirmation status. This is misleading.
</Warning>

### How to Use

<Tabs>
  <Tab title="Confirm a Pending Booking">
    ```bash
    # DELETE request to confirm booking with ID 5
    DELETE /bookings/publish/5
    ```
    
    If `status = 0`, it will be set to `1` (confirmed)
  </Tab>
  
  <Tab title="Unconfirm a Booking">
    ```bash
    # DELETE request to unconfirm booking with ID 5
    DELETE /bookings/publish/5
    ```
    
    If `status = 1`, it will be set to `0` (pending)
  </Tab>
</Tabs>

<Note>
  The confirm endpoint toggles the status. Check the current status before calling this endpoint.
</Note>

## Deleting a Booking

**Route:** `/bookings/delete/{id}`  
**Method:** DELETE  
**Controller Method:** `destroy(Booking $booking, $id)`  
*Source: BookingController.php:158-164*

```php
public function destroy(Booking $booking, $id)
{
    $booking = Booking::find($id);
    $booking->delete();
    return redirect()->route('bookings')
        ->with('success', 'Booking deleted successfully!');
}
```

<Warning>
  **Deletion is permanent!** There is no soft delete implemented. Once a booking is deleted, it cannot be recovered.
</Warning>

<Steps>
  <Step title="Locate the Booking">
    Find the booking you want to delete in the bookings list at `/bookings`
  </Step>
  
  <Step title="Confirm Deletion">
    Click the delete button (should show confirmation dialog)
  </Step>
  
  <Step title="Booking Removed">
    The booking is permanently deleted from the database and you're redirected to the bookings list
  </Step>
</Steps>

## Booking Workflow Diagram

<Steps>
  <Step title="Customer Creates Booking">
    Customer fills out booking form at `/bookings/create/{destination}`
    - Must complete reCAPTCHA
    - Must provide matching email addresses
    - Must include payment transaction code
  </Step>
  
  <Step title="Booking Submitted">
    Booking is stored with `status = 0` (pending)
    - Customer receives success message
    - Redirected to profile page
    - Admin can view in bookings list
  </Step>
  
  <Step title="Admin Reviews Booking">
    Admin accesses `/bookings` to view all pending bookings
    - Can view full booking details
    - Can verify payment transaction code
    - Can check availability
  </Step>
  
  <Step title="Admin Confirms Booking">
    Admin clicks confirm button (`/bookings/publish/{id}`)
    - Status changes to `status = 1` (confirmed)
    - Customer should be notified (manual process)
  </Step>
  
  <Step title="Optional: Edit or Delete">
    Admin can:
    - Edit booking details if needed
    - Delete booking if it's fraudulent or cancelled
    - Unconfirm if there's an issue
  </Step>
</Steps>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Email Mismatch Error" icon="envelope">
    **Error:** "please provide matching emails"
    
    **Cause:** email1 and email2 fields don't match
    
    **Solution:**
    - User must enter exact same email twice
    - Check for typos, spaces, or case differences
    - Ensure both fields are properly filled
  </Accordion>
  
  <Accordion title="reCAPTCHA Validation Failed" icon="shield-alt">
    **Error:** Validation fails on g-recaptcha-response
    
    **Cause:** reCAPTCHA not completed or expired
    
    **Solution:**
    - Ensure user completes reCAPTCHA challenge
    - Check if reCAPTCHA keys are configured in `.env`
    - Verify reCAPTCHA domain is whitelisted
  </Accordion>
  
  <Accordion title="Cannot Access Bookings" icon="lock">
    **Requirements:**
    - Must be logged in (`auth` middleware)
    - Email must be verified (`verified` middleware)
    
    **Solution:**
    - Log in to your account
    - Verify your email if you haven't already
    - Check that your account has proper permissions
  </Accordion>
  
  <Accordion title="Redirect Error After Update" icon="exclamation-triangle">
    **Error:** Route [tours.bookings] not defined
    
    **Cause:** The redirect route name in update method is incorrect
    
    **Solution:**
    - This is a bug in the code (line 149 of BookingController.php)
    - Manually change redirect to: `redirect()->route('bookings')`
  </Accordion>
  
  <Accordion title="Status Not Changing" icon="toggle-on">
    **Issue:** Clicking confirm doesn't change status
    
    **Possible causes:**
    - JavaScript not properly sending DELETE request
    - CSRF token missing or invalid
    - Middleware blocking request
    
    **Solution:**
    - Check browser console for errors
    - Ensure form includes `@csrf` token
    - Verify user is authenticated and verified
  </Accordion>
</AccordionGroup>

## User Relationship

```php

{
    return $this->hasMany('App\\Booking');
```

## Code Issues and Recommendations

<AccordionGroup>
  <Accordion title="Incorrect HTTP Methods" icon="exclamation-circle">
    **Issue:** Confirm route uses DELETE method
    
    ```php
    // Current (incorrect)
    Route::delete('/bookings/publish/{id}', ...)
    
    // Should be
    Route::put('/bookings/publish/{id}', ...)
    ```
    
    **Impact:** Semantically incorrect - DELETE should remove resources, not update them
  </Accordion>
  
  <Accordion title="Misleading Success Messages" icon="comment">
    **Issue:** Confirm method shows "Booking deleted successfully!" but actually confirms/unconfirms
    
    ```php
    // Current (misleading)
    ->with('success', 'Booking deleted successfully!');
    
    // Should be
    ->with('success', 'Booking status updated!');
    ```
  </Accordion>
  
  <Accordion title="Invalid Route Name" icon="route">
    **Issue:** Update method redirects to non-existent route
    
    ```php
    // Current (broken)
    return redirect()->route('tours.bookings')
    
    // Should be
    return redirect()->route('bookings')
    ```
  </Accordion>
  
  <Accordion title="Missing Soft Deletes" icon="trash-restore">
    **Issue:** Bookings are permanently deleted with no recovery option
    
    **Recommendation:** Implement soft deletes:
    
    ```php
    use Illuminate\Database\Eloquent\SoftDeletes;
    
    class Booking extends Model {
        use SoftDeletes;
    }
    ```
  </Accordion>
</AccordionGroup>
